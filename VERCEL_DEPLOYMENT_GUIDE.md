# Vercel Serverless Backend Deployment Guide

## Overview
This guide explains how the campus-lost-found application is configured for Vercel serverless deployment with a separate API backend and frontend.

## Architecture

### Folder Structure
```
project-root/
├── api/                           # Vercel serverless API handler
│   ├── index.js                  # Main API handler (Express app wrapper)
│   └── [...all].ts               # Catch-all route for Vercel
│
├── backend/                       # Backend Express application (TypeScript)
│   ├── src/
│   │   ├── app.ts               # Express app configuration
│   │   ├── index.ts             # Entry point (local dev) / exports app
│   │   ├── serverless.ts        # Serverless initialization (DB, Firebase)
│   │   ├── setup.ts             # Environment setup
│   │   ├── config/
│   │   │   ├── database.ts      # MongoDB connection
│   │   │   └── firebase.ts      # Firebase initialization
│   │   ├── routes/              # API route handlers
│   │   ├── middleware/          # Express middleware
│   │   └── services/            # Business logic
│   ├── dist/                    # Compiled JavaScript (generated by TypeScript)
│   ├── package.json
│   ├── tsconfig.json
│   └── ...

├── frontend/                      # Vue/React frontend
│   └── src/

│       └── ...

```

## How It Works

### 1. Request Flow
```
Browser Request → /api/items
    ↓
Vercel Router (vercel.json)
    ↓
Route Rule: /api/(.*) → /api/index.js
    ↓
api/index.js handler function
    ↓
Initialize serverless (DB, Firebase) on first request
    ↓
Express app processes request
    ↓
Responds to browser
```

### 2. Local Development
- Backend runs on `http://localhost:3001`
- Frontend runs on `http://localhost:5173` (Vite)
- Frontend makes requests to `http://localhost:3001/api` by default
- `backend/src/index.ts` includes `app.listen(3001)` when `!process.env.VERCEL`

### 3. Vercel Deployment
- Backend code (TypeScript) is compiled to `backend/dist/` JavaScript
- `api/index.js` imports the compiled app and handles requests
- `vercel.json` tells Vercel how to route requests
- Serverless functions are stateless but use caching to reduce DB initialization overhead
- Frontend and backend are deployed to the same Vercel project for CORS to work via relative paths

## Configuration Files

### vercel.json
```json
{
  "version": 2,
  "installCommand": "NODE_OPTIONS='--no-deprecation' pnpm install",
  "buildCommand": "pnpm run build",
  "outputDirectory": "dist",
  "env": {
    "NODE_OPTIONS": "--no-deprecation"
  },
  "builds": [
    {
      "src": "api/index.js",
      "use": "@vercel/node"
    }
  ],
  "routes": [
    {
      "src": "/api/(.*)",
      "dest": "/api/index.js"
    },
    {
      "src": "^/(assets/.*|favicon\\.ico|robots\\.txt|sitemap\\.xml)$",
      "dest": "/$1"
    },
    {
      "src": "/(.*)",
      "dest": "/index.html"
    }
  ]
}
```

**Key Points:**
- `"src": "api/index.js", "use": "@vercel/node"` - Tells Vercel this is a Node.js serverless function
- Route rule `/api/(.*)` → `/api/index.js` - All API requests go to the Express handler
- Route rule `/(.*)`  → `/index.html` - SPA fallback for frontend

## Key Code Files

### api/index.js (Vercel Handler)
```javascript
import app from '../backend/dist/app.js';
import { initializeServerless } from '../backend/dist/serverless.js';

// Initialize once at module load (cold start)
let initialized = false;
let initPromise = null;

async function ensureInitialized() {
  if (initialized) return;
  
  if (initPromise) {
    return initPromise;
  }

  initPromise = initializeServerless()
    .then(() => {
      initialized = true;
      console.log('[Vercel API] Serverless backend initialized');
    })
    .catch((error) => {
      console.error('[Vercel API] Failed to initialize:', error);
      initPromise = null; // Reset for retry
      throw error;
    });

  return initPromise;
}

// Export handler for Vercel
export default async function handler(req, res) {
  try {
    await ensureInitialized();
  } catch (error) {
    console.error('[API Handler] Initialization failed:', error);
    return res.status(500).json({ 
      error: 'Backend initialization failed',
      message: error.message 
    });
  }

  // Pass request through Express app
  return app(req, res);
}
```

### backend/src/serverless.ts
```typescript
import { connectDB } from './config/database.js';
import { initializeFirebase } from './config/firebase.js';

let initialized = false;
let initPromise: Promise<void> | null = null;

export async function initializeServerless(): Promise<void> {
  if (initialized) {
    return;
  }

  if (initPromise) {
    return initPromise;
  }

  initPromise = (async () => {
    console.log('[Serverless] initializing database and firebase');

    await connectDB();
    console.log('[Serverless] database connected');

    initializeFirebase();
    console.log('[Serverless] firebase initialized');

    initialized = true;
  })();

  return initPromise;
}
```

### backend/src/index.ts
```typescript
import app from './app.js';
import { initializeServerless } from './serverless.js';

const PORT = process.env.PORT || 3001;

async function bootstrap(): Promise<void> {
  try {
    console.log('[Backend] Bootstrapping starting...');

    await initializeServerless();

    if (!process.env.VERCEL) {
      app.listen(PORT, () => {
        console.log(`[Backend] Server running on port ${PORT}`);
        console.log(`[Backend] Environment: ${process.env.NODE_ENV || 'development'}`);
      });
    } else {
      console.log('[Backend] Running in Vercel serverless mode; no permanent listener started.');
    }
  } catch (error) {
    console.error('[Backend] Bootstrapping failed:', error);
    if (!process.env.VERCEL) {
      process.exit(1);
    }
  }
}

if (!process.env.VERCEL) {
  bootstrap();
}

export default app;
```

### backend/src/app.ts (Express Setup)
The app.ts file:
- Creates the Express application
- Sets up helmet, CORS, and security middleware
- Defines the `/api/health` endpoint for testing
- Registers all route handlers (`/api/items`, `/api/auth`, etc.)
- Exports the Express app object

**Does NOT call:**
- `app.listen()` - This is handled in index.ts (local) or api/index.js (Vercel)
- `connectDB()` or `initializeFirebase()` - These are called by serverless.ts

## Deployment Checklist

### Before Deploying to Vercel:

1. **Build Locally**
   ```bash
   cd backend
   npm run build
   cd ..
   npm run build
   ```
   This creates `backend/dist/` with compiled JavaScript.

2. **Verify Files**
   - [ ] `backend/dist/app.js` exists
   - [ ] `backend/dist/serverless.js` exists
   - [ ] `api/index.js` exists and is updated
   - [ ] `vercel.json` is present
   - [ ] `.env.local` has all required variables for local testing

3. **Test Locally**
   ```bash
   # Terminal 1: Start backend
   cd backend
   npm run dev
   
   # Terminal 2: Start frontend
   npm run dev
   ```
   - Frontend: visit `http://localhost:5173`
   - Backend health: visit `http://localhost:3001/api/health`
   - API: `http://localhost:3001/api/items`, etc.

4. **Push to Git**
   ```bash
   git add .
   git commit -m "Fix Vercel serverless backend deployment"
   git push
   ```

5. **Deploy to Vercel**
   - Connect your repository to Vercel (if not already)
   - Vercel will automatically:
     1. Run `pnpm install`
     2. Run `pnpm run build` (builds backend TypeScript and frontend)
     3. Upload compiled files
     4. Detect `api/index.js` as serverless function
   - Set environment variables in Vercel dashboard

## Common Issues & Solutions

### Issue 1: "Raw JavaScript Source Returned"
**Cause:** `api/index.js` is being served as static file instead of executed as function.

**Solution:**
- Ensure `vercel.json` exists with correct `"builds"` configuration
- Ensure `api/index.js` has proper `export default function handler(req, res)`
- Run `npm run build` to compile backend TypeScript

### Issue 2: 404 on `/api/items`
**Cause:** Routes not being recognized.

**Solution:**
- Check vercel.json routing rules
- Verify `api/index.js` is importing the correct app path
- Check Express routes in `backend/src/app.ts`

### Issue 3: CORS Errors
**Cause:** Frontend and backend on different domains without proper CORS setup.

**Solution:**
- Set `FRONTEND_URL` environment variable in Vercel
- Update `backend/src/app.ts` CORS configuration to allow the frontend URL
- For same-domain deployments (recommended), both frontend and backend are deployed to same Vercel project

### Issue 4: Database Connection Timeout
**Cause:** MongoDB connection fails.

**Solution:**
- Verify `MONGODB_URI` is correct and set in Vercel environment
- Check MongoDB IP allowlist includes Vercel's IP ranges
- Check connection string has correct username/password

### Issue 5: Firebase Initialization Error
**Cause:** Firebase credentials not set or invalid.

**Solution:**
- Verify all `FIREBASE_*` variables are set in Vercel dashboard
- Private key must have proper formatting with newlines (`\n`)
- Test locally first with `.env.local`

## Performance Notes

1. **Cold Start**: First request after deployment takes longer (DB+Firebase init)
2. **Warm Start**: Subsequent requests reuse cached connections
3. **Connection Pooling**: MongoDB uses connection pooling to minimize overhead
4. **Global Cache**: Uses `globalThis._mongoConnection` to cache connections across function invocations
